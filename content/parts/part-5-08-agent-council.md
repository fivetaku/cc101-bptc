# Part 5 / Clip 8: Agent Council로 설계 검증

> 강의 영상: Part 5 / Clip 8 (~25분)
> 만드는 것: Council 검토 보고서 + 반영 결과

## 🎯 이 클립에서 만드는 것

에이전트를 하나 만들고 나면 기분이 좋습니다. 그런데 여기서 바로 "완성!"이라고 하기엔 살짝 이릅니다. 내가 만든 설계는 내 설명을 잘 따른 결과라서, 내 blind spot도 그대로 따라올 수 있거든요.

이번 클립에서는 Workspace Builder 안의 Agent Council을 소집합니다. Claude가 만든 Phase 4 설계를 Codex와 Gemini 관점으로 다시 보고, Chairman이 종합 의견을 냅니다. 말하자면 생성 전 설계 리뷰 회의예요.

| Before | After |
|---|---|
| Claude와 나만 보고 설계 확정 | Codex, Gemini, Chairman 종합 의견까지 확인 |
| "괜찮아 보이는데?"에서 멈춤 | 범위, 커맨드, 검증, 보안 관점으로 다시 점검 |
| 수정할지 말지 감으로 결정 | Council 권고 중 반영 / 보류를 나눠 기록 |
| 검토 내용이 대화창에 흩어짐 | `council-review.md`와 반영 결과로 남김 |

Codex CLI와 Gemini CLI가 설치되어 있으면 같이 부릅니다. 없으면 괜찮습니다. Claude만으로도 설계 검토를 진행할 수 있어요. 중요한 건 "한 번 멈춰서 검토한다"는 습관입니다.

## 💡 핵심 개념

### Council은 AI 회의실입니다

Agent Council은 다른 AI에게 설계를 맡겨보는 장치입니다. Claude가 설계자라면, Codex는 구현 가능성과 구조를 더 빡세게 볼 수 있고, Gemini는 다른 관점의 누락을 짚을 수 있습니다. 마지막에 Chairman이 의견을 묶어 "무엇을 반영할지" 정리해 줍니다.

| 역할 | 주로 보는 것 |
|---|---|
| Claude | 사용자의 목적, 대화 맥락, 최종 설계 요약 |
| Codex | 커맨드 구조, 실행 가능성, 파일 구성, 검증 가능성 |
| Gemini | 범위 누락, 사용자 경험, 다른 해석 가능성 |
| Chairman | 충돌하는 의견을 묶고 반영 우선순위 제안 |

이건 AI가 AI를 대신 판단해주는 마법이 아닙니다. 여러분이 결정하기 전에 시야를 넓히는 과정이에요. 회의에서 다른 부서 사람이 "그거 실제 운영하면 누가 입력해요?"라고 묻는 순간 있잖아요. 그런 질문을 얻는 게 목표입니다.

### Phase 4에서 부르는 게 가장 좋습니다

Workspace Builder 문서에서는 Agent Council을 Phase 2, Phase 3, Phase 4에서 쓸 수 있다고 안내합니다. 이번 강의에서는 Phase 4, 즉 생성 직전 최종 설계 검토에 씁니다. 이유는 간단해요. 설계가 어느 정도 모였을 때 봐야 의견이 구체적으로 나옵니다.

| 시점 | Council 용도 | 이번 실습 |
|---|---|---|
| Phase 2 | 아키타입 선택 검증 | 선택 |
| Phase 3 | 워크플로우 설계 검토 | 선택 |
| Phase 4 | 최종 설계 검토 | 권장 |

오늘은 `council 소집해줘` 한 문장으로 시작합니다. 결과를 보고 "전부 반영"이 아니라, "반영할 것 / 보류할 것"을 고르는 데 집중하세요.

## 📁 시작 전 — 자료 셋업

먼저 Clip 6에서 만든 Workspace Builder 대화 또는 생성된 설계가 있어야 합니다. 가능하면 Phase 4 설계 요약이 남아 있는 상태에서 진행하세요. 이미 생성까지 끝냈다면, 생성된 워크스페이스의 `CLAUDE.md`와 커맨드 파일을 기준으로 검토해도 됩니다.

### 1단계. Codex와 Gemini CLI 확인

터미널에서 아래 명령을 실행합니다.

```bash
command -v codex && command -v gemini
```

둘 다 경로가 나오면 Council에서 다른 AI 의견을 함께 받을 수 있습니다. 하나라도 없으면 너무 당황하지 마세요. 오늘 실습은 Claude 단독 검토로도 진행 가능합니다.

### 2단계. Workspace Builder로 이동

```bash
cd ~/Desktop/Workspace_Builder
claude
```

Clip 6 대화가 이어져 있으면 그 상태에서 진행합니다. 새로 열었다면 생성된 에이전트 위치를 알려주면 됩니다.

```text
지난 Clip 6에서 만든 에이전트는 ~/Desktop/Workspace_Builder/90-output/{내에이전트}/ 입니다. 이 설계를 Council로 검토하고 싶습니다.
```

### 3단계. 실습 기록 폴더 준비

```text
~/Desktop/bptc-cc/50-my-work/Part05-에이전트구축/실습27-Council검증/ 폴더를 만들고, Council 검토 결과를 council-review.md로 정리할 준비를 해 주세요.
```

### 셋업 완료 체크

## 🚶 진행 흐름

### 1. 검토 대상 지정하기

Council을 부르기 전에 무엇을 검토할지 지정합니다. 그냥 "검토해줘"보다 훨씬 낫습니다.

```text
Clip 6에서 만든 [에이전트 이름] 설계를 검토하려고 합니다. 검토 대상은 1) 아키타입 선택, 2) 핵심 커맨드, 3) 첫 버전 범위, 4) 생성된 폴더 구조, 5) 실제 업무 적용 가능성입니다. 먼저 현재 설계를 요약해 주세요.
```

요약이 맞는지 확인하세요. 설계 요약이 틀리면 Council도 틀린 걸 보고 의견을 냅니다.

### 2. Council 소집하기

이제 Workspace Builder 안에서 Council을 부릅니다.

```text
council 소집해줘
```

또는 조금 더 구체적으로 부릅니다.

```text
council 소집해줘. 이 에이전트 설계를 Codex와 Gemini 관점에서 검토하고, Chairman 종합 의견까지 정리해줘.
```

CLI가 설치되어 있다면 Codex 의견, Gemini 의견이 따로 나옵니다. 설치되어 있지 않다면 Claude가 현재 정보만으로 검토를 진행할 수 있습니다.

### 3. 의견을 세 갈래로 읽기

Council 결과가 나오면 한 번에 다 고치려고 하지 마세요. 먼저 의견을 나눕니다.

```text
Council 결과를 세 갈래로 나눠 주세요. 1) 반드시 반영할 것, 2) 이번 버전에서는 보류할 것, 3) 제가 결정해야 할 것. 각 항목은 이유를 한 줄씩만 붙여 주세요.
```

여기서 저는 "반드시 반영"이 2~3개를 넘지 않게 보는 편입니다. 첫 버전부터 너무 많이 고치면 에이전트가 다시 커집니다.

### 4. 반영할 수정만 적용하기

반영 항목을 고른 뒤 수정합니다. 이미 워크스페이스가 생성되어 있다면 `CLAUDE.md`, 커맨드 설명, README 정도가 수정 대상이 될 수 있습니다.

```text
Council 권고 중 아래 항목만 반영해 주세요.

- [반영 항목 1]
- [반영 항목 2]

수정 범위는 생성된 워크스페이스의 CLAUDE.md와 관련 커맨드 설명으로 제한하고, 수정 후 어떤 파일이 바뀌었는지 요약해 주세요.
```

아직 생성 전이라면 Phase 4 설계에 반영한 뒤 Phase 5 생성으로 넘어가면 됩니다.

### 5. council-review.md 작성하기

마지막은 기록입니다. Council은 검토 회의라서, 회의록이 없으면 다음에 왜 바꿨는지 기억이 안 납니다.

```text
Council 검토 결과를 ~/Desktop/bptc-cc/50-my-work/Part05-에이전트구축/실습27-Council검증/council-review.md로 정리해 주세요. 포함할 내용은 1) 검토 대상 에이전트, 2) CLI 확인 결과, 3) Codex 의견, 4) Gemini 의견, 5) Chairman 종합, 6) 반영한 항목, 7) 보류한 항목입니다.
```

## 📦 결과물

저장 위치는 `50-my-work/Part05-에이전트구축/실습27-Council검증/`입니다.

```text
실습27-Council검증/
├── council-review.md
└── README.md
```

`council-review.md`에는 Council 검토 내용을 남깁니다.

- 검토 대상 에이전트 이름과 위치
- `command -v codex && command -v gemini` 확인 결과
- Codex 의견 요약
- Gemini 의견 요약
- Chairman 종합 의견
- 실제 반영한 수정
- 이번 버전에서 보류한 수정

`README.md`에는 반영 후 에이전트가 어떻게 달라졌는지 짧게 적습니다. 예를 들어 "커맨드 6개를 4개로 줄였다", "입력 파일 조건을 README에 추가했다", "민감정보는 샘플 데이터로만 테스트하도록 바꿨다"처럼요.

## 🆘 자주 막히는 포인트

막히는 데는 보통 정해져 있어요. Council은 설치 상태와 검토 범위에서 많이 흔들립니다.

| 증상 | 원인 | 해결 |
|---|---|---|
| `codex` 경로가 안 나옴 | Codex CLI가 설치되어 있지 않음 | Claude 단독 검토로 진행해도 됩니다 |
| `gemini` 경로가 안 나옴 | Gemini CLI가 설치되어 있지 않음 | Gemini 의견은 선택으로 두면 돼요 |
| Council이 뭘 봐야 할지 모름 | 검토 대상이 흐림 | 에이전트 이름, 위치, 검토 항목 5개를 먼저 줍니다 |
| 의견이 너무 많음 | 전체를 다 고치려고 함 | 반드시 반영 2~3개만 고르세요 |
| 서로 다른 의견이 충돌함 | AI별 관점이 다름 | Chairman에게 우선순위를 정리해달라고 하면 됩니다 |
| 생성된 파일을 못 찾음 | `90-output` 위치를 놓침 | `ls ~/Desktop/Workspace_Builder/90-output`으로 확인합니다 |
| 보고서가 길어짐 | 원문을 전부 복사함 | 의견 요약, 반영, 보류만 남기면 충분해요 |

여전히 애매하면 "이번 버전에서 반영하지 않아도 되는 건 뭐야?"라고 물어보세요. 좋은 검토는 고칠 것만큼 안 고칠 것도 정해줍니다.
