## 🎯 이 클립에서 만드는 것

여기까지 오면 사실 꽤 많이 온 겁니다. 대화하고, 파일 만들고, 배포하고, 플러그인 쓰고, PRD 만들고, 마지막에는 내 업무용 에이전트 워크스페이스까지 만들었죠. 이제 필요한 건 하나입니다. "그래서 내가 뭘 얻었고, 다음에 어떻게 보여줄 건가?"

이번 클립은 Part 5의 마지막입니다. 1박2일 워크샵을 회고하고, 6회차 발표에서 내가 만든 에이전트를 어떻게 시연할지 정리합니다. 그리고 BPTC 14명이 각자 만든 에이전트를 사내에서 어떻게 공유하고 발전시킬지도 계획해요.

| Before | After |
|---|---|
| 실습 결과물이 여러 폴더에 흩어짐 | 핵심 결과와 배운 점을 회고 노트로 정리 |
| 만든 에이전트를 어떻게 보여줄지 막막함 | 3분 시연 흐름과 백업 설명 준비 |
| 개인 실습으로 끝남 | 14명 에이전트를 부서별로 공유할 확산 계획 작성 |
| 다음 회차에 뭘 가져갈지 애매함 | 발표 자료, 시연 경로, 개선 질문을 정리 |

결과물은 회고 노트와 발표 준비 문서입니다. 화려한 보고서가 아니라, 다음 회차에 바로 열어볼 수 있는 메모면 충분합니다.

## 💡 핵심 개념

### 회고는 감상이 아니라 다음 행동을 고르는 일입니다

회고라고 하면 "좋았다, 어려웠다"로 끝나기 쉽습니다. 그런데 워크샵 회고는 조금 달라야 합니다. 가장 임팩트 있던 클립은 왜 좋았는지, 막혔던 부분은 어디서 막혔는지, 다음 회차에 무엇을 가져갈지까지 정해야 해요.

| 회고 항목 | 적는 내용 |
|---|---|
| 가장 임팩트 있던 클립 | 왜 기억에 남았는지, 내 업무와 어디가 연결됐는지 |
| 막혔던 부분 | 경로, 설치, 프롬프트, 개념 중 어디서 막혔는지 |
| 다시 해볼 부분 | 시간이 부족해서 덜 한 실습이나 고치고 싶은 결과물 |
| 다음 회차에 가져갈 것 | 발표할 에이전트, 질문, 개선 요청 |

저는 회고를 "내가 다음에 덜 헤매기 위한 메모"라고 봅니다. 예쁘게 쓰는 게 목적이 아니에요. 다음 회차에 열었을 때 바로 기억나면 됩니다.

### 발표는 완성품 자랑보다 흐름 시연입니다

6회차 발표에서 중요한 건 "내 에이전트가 모든 걸 다 합니다"가 아닙니다. 문제를 어떻게 잡았고, 어떤 아키타입을 골랐고, 어떤 커맨드로 첫 작업을 시킬 수 있는지 보여주는 겁니다.

| 발표 흐름 | 내용 |
|---|---|
| 1. 문제 | 내 업무에서 반복되거나 귀찮았던 일 |
| 2. 에이전트 | 만든 워크스페이스 이름과 아키타입 |
| 3. 시연 | 생성된 폴더에서 `claude` 실행 후 대표 커맨드 1개 |
| 4. 결과 | 어떤 파일이나 요약이 나오는지 |
| 5. 다음 개선 | Council에서 받은 피드백 또는 앞으로 붙일 기능 |

시연은 짧아야 합니다. 3분 안에 보여줄 수 있어야 해요. 시간이 길어지는 순간 보는 사람은 "좋은데 우리 업무에 어떻게 쓰지?"를 놓칩니다.

### 14명의 에이전트는 작은 사내 도구 목록이 됩니다

BPTC 14명이 각자 하나씩 만들면, 단순히 실습 결과물이 14개 생기는 게 아닙니다. 부서별 작은 업무 비서 목록이 생기는 거예요. 누군가는 자재관리, 누군가는 보고서, 누군가는 교육, 누군가는 리서치. 이걸 공유하면 서로의 아이디어가 빨리 번집니다.

| 공유 방식 | 좋은 점 |
|---|---|
| 에이전트 이름 + 한 줄 설명 목록 | 빠르게 훑어볼 수 있음 |
| 부서별 분류 | 비슷한 업무끼리 묶기 쉬움 |
| 대표 커맨드 1개 공유 | 실제 사용 장면이 보임 |
| 개선 요청 칸 | 다음 버전 아이디어가 쌓임 |

오늘은 이 확산 계획까지 적습니다. 워크샵이 끝나는 순간이 아니라, 사내에서 시작되는 순간으로 만들기 위해서요.

## 🚶 진행 흐름

### 1. 1박2일 워크샵 회고하기

먼저 전체를 돌아봅니다. 길게 쓰지 말고, 기억나는 장면부터 꺼내세요.

```text
1박2일 워크샵 회고 노트를 작성하려고 합니다. 항목은 1) 가장 임팩트 있던 클립, 2) 막혔던 부분, 3) 내가 만든 결과물 중 가장 쓸 만한 것, 4) 다음 회차에 가져갈 것입니다. 질문을 하나씩 던져가며 제 답을 받아 retrospect.md로 정리해 주세요.
```

막혔던 부분을 적을 때는 부끄러워할 필요 없습니다. 경로에서 막혔는지, 설치에서 막혔는지, 프롬프트에서 막혔는지 남겨야 다음 운영이 좋아집니다.

### 2. 만든 에이전트 한 줄 소개 만들기

발표 첫 문장은 짧아야 합니다. 듣는 사람이 바로 이해해야 해요.

```text
제가 만든 에이전트 워크스페이스를 6회차 발표에서 소개하려고 합니다. 에이전트 이름, 해결하려는 업무, 선택한 아키타입, 대표 커맨드 1개를 바탕으로 15초 소개 문장을 3가지 버전으로 만들어 주세요.
```

예시는 이런 느낌입니다.

```text
제가 만든 것은 BPTC 자재관리 에이전트입니다. 주간 입출고 자료를 읽고, 확인이 필요한 품목과 회의용 요약을 만들어주는 Knowledge Base형 업무 비서입니다.
```

### 3. 3분 시연 흐름 짜기

시연은 짧게 갑니다. 폴더 위치를 보여주고, `claude`를 실행하고, 대표 커맨드 하나만 보여주세요.

```text
6회차 발표용 3분 시연 흐름을 작성해 주세요. 포함할 내용은 1) 문제 소개 30초, 2) 생성된 에이전트 폴더 소개 30초, 3) 대표 커맨드 실행 1분, 4) 결과물 확인 40초, 5) 다음 개선 20초입니다.
```

시연 중 실제 데이터가 민감하면 샘플 데이터로 바꿉니다. 회사 자료를 그대로 띄우지 않아도 됩니다. 핵심은 작동 흐름을 보여주는 거예요.

### 4. 백업 설명 준비하기

라이브 시연은 언제든 흔들릴 수 있습니다. 그래서 백업 설명이 필요합니다.

```text
라이브 시연이 실패했을 때 대신 설명할 백업 스크립트를 만들어 주세요. 터미널이 안 열리거나 Claude 응답이 늦어져도, 생성된 파일 구조와 대표 커맨드 설계만으로 설명할 수 있게 구성해 주세요.
```

이걸 준비해두면 발표 때 마음이 편합니다. 실습 발표는 완벽한 데모보다 차분한 설명이 더 중요할 때가 많아요.

### 5. BPTC 14명 사내 확산 계획 작성하기

마지막은 개인 결과물을 팀 자산으로 바꾸는 계획입니다. 14명이 만든 에이전트를 한 번에 다 운영하려고 하지 말고, 목록화하고, 비슷한 것끼리 묶고, 대표 사례를 고릅니다.

```text
BPTC 14명이 각자 만든 에이전트 워크스페이스를 사내에 공유하고 발전시키는 계획을 작성하려고 합니다. 1) 에이전트 목록 수집 양식, 2) 부서별 분류 방식, 3) 월 1회 공유 방식, 4) 좋은 에이전트 선정 기준, 5) 다음 버전 개선 루프를 포함해 spread-plan.md로 정리해 주세요.
```

여기서 "확산"은 거창한 시스템 도입이 아닙니다. 누가 어떤 비서를 만들었는지 서로 알게 하는 것부터 시작합니다.

### 6. README로 마무리하기

마지막에는 세 문서를 묶어 README를 만듭니다.

```text
실습28-회고/README.md를 작성해 주세요. retrospect.md, demo-plan.md, spread-plan.md의 핵심을 요약하고, 6회차 발표 전 체크리스트 5개를 넣어 주세요.
```

## 🆘 자주 막히는 포인트

막히는 데는 보통 정해져 있어요. 마지막 클립은 기술보다 정리가 어렵습니다.

| 증상 | 원인 | 해결 |
|---|---|---|
| 회고가 감상문처럼 됨 | 다음 행동이 없음 | "다음 회차에 가져갈 것"을 마지막에 꼭 적으면 돼요 |
| 발표가 너무 김 | 만든 과정을 전부 설명함 | 문제, 에이전트, 대표 커맨드, 결과만 남기세요 |
| 시연이 불안함 | 라이브 실행만 믿음 | 백업 스크립트와 파일 구조 설명을 준비하면 됩니다 |
| 회사 자료를 보여줘야 할 것 같음 | 실제 데이터와 시연 데이터를 구분 안 함 | 샘플 데이터로 흐름만 보여줘도 충분해요 |
| 확산 계획이 거창해짐 | 시스템 도입처럼 생각함 | 14명 에이전트 목록 공유부터 시작하면 됩니다 |
| 좋은 에이전트 기준이 모호함 | 이름만 보고 판단함 | 반복성, 입력 명확성, 결과물 재사용성을 기준으로 보세요 |
| README가 길어짐 | 세 문서를 다 복사함 | 발표 전 체크리스트 5개만 선명하게 남기면 돼요 |

여전히 막히면 "내 발표를 3분으로 줄이면 무엇을 빼야 해?"라고 물어보세요. 발표는 덜어낼수록 좋아집니다.
